<objective>
Neutralize the milestone's central risk. The site currently emits an UNCONDITIONAL `noindex` robots meta (src/routes/+layout.svelte line 15) and an unconditional `robots.txt` `Disallow: /` (static/robots.txt). Both are correct for the staging github.io host but FATAL for the apex — they would hide the live michellengo.net from search engines.

This plan environment-gates BOTH off the existing BASE_PATH signal that already differentiates the two deploy workflows:
- Staging workflow (deploy.yml) builds with `BASE_PATH: /michelle_ngo_five` and must KEEP `noindex` + `Disallow: /` (unchanged behavior).
- Production workflow (deploy-production.yml) builds with `BASE_PATH: ''` and must become crawlable (NO `noindex`, robots.txt `Allow: /`).

Then it documents the indexing-gate verification step in CUTOVER.md so Phase 6 can confirm the gate took effect in production.

Purpose: Make the apex build search-indexable without changing staging behavior — the precondition for the Phase 6 DNS flip.
Output: A conditional robots meta in the layout, a prerendered env-gated robots.txt route, removal of the static robots.txt, and a CUTOVER.md verification step.
</objective>

         });
