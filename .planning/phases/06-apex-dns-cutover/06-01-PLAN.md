<objective>
Pre-flight the apex cutover and flip DNS for michellengo.net from the prior WordPress.com site to
the v5 GitHub Pages build. This plan performs the WordPress.com-specific addendum the runbook lacks
(primary-site-address move, name-server confirm, CAA inspect/clear, MX/TXT snapshot), publishes the
base-'' production apex build, then replaces the apex A/AAAA records with GitHub's 8 IPs and points
www at wolfwdavid.github.io — leaving MX/TXT untouched.

This plan STOPS once DNS resolves to the four GitHub IPs. Registering the domain in repo Settings,
provisioning TLS, and running the six-check done-gate are Plan 06-02.

Purpose: Execute CUTOVER.md Steps 1–2 (plus the research-mandated WP.com addendum) safely, so the
apex points at a styled, indexable, root-relative build with email intact.
Output: michellengo.net resolving to GitHub Pages IPs; a green apex deploy in the github-pages
environment; MX/TXT confirmed preserved; no CAA blocking Let's Encrypt.
</objective>

  <action>
    This is a HUMAN-ONLY step. Claude cannot log into WordPress.com — coordinate with the account
    holder and confirm each item is done before resuming. Do these IN ORDER, in WordPress.com:

    (a) MOVE THE PRIMARY SITE ADDRESS OFF THE APEX (so old-site admin survives the flip):
        WordPress.com → Settings → General → Site Address — change the primary site address to a
        `*.wordpress.com` (or `*.wpcomstaging.com`) address. If michellengo.net is currently the
        primary domain of the WP.com site, set a wordpress.com subdomain as primary instead.
        WHY: once the apex resolves to GitHub (Task 3), the michellengo.net admin URL is unreachable;
        the *.wordpress.com address keeps the old dashboard accessible. (RESEARCH Pitfall 1.)

    (b) CONFIRM NAME SERVERS REMAIN AT WORDPRESS.COM:
        WordPress.com → Upgrades → Domains → michellengo.net → confirm the name servers are still the
        WordPress.com / Automattic name servers (NOT changed to GitHub or anyone else). The DNS editor
        only controls the zone if NS stay at WP.com. This is REQUIRED for Task 3's edits to take effect.

    (c) DO NOT DETACH / DISCONNECT / DELETE THE DOMAIN. No detach is required (D-02, RESEARCH Pitfall 1).
        The verb is "EDIT DNS RECORDS," never "detach" or "delete the domain" — Delete severs the
        connection permanently. Adding custom A records auto-replaces the "Handled by WordPress.com"
        defaults; that is the correct and sufficient mechanism.

    (d) INSPECT AND CLEAR A BLOCKING CAA RECORD:
        WordPress.com → Upgrades → Domains → michellengo.net → DNS records. Look for any `CAA` record.
        If a CAA record exists and does NOT include `letsencrypt.org`, either DELETE it (no CAA = any
        CA allowed) or ADD a CAA record `0 issue "letsencrypt.org"`. CAA is a DISTINCT record type from
        MX/TXT — adjusting CAA does NOT affect email and does NOT violate D-05. WHY: a CAA that omits
        letsencrypt.org silently blocks GitHub's Let's Encrypt cert for hours (RESEARCH Pitfall 2).

    (e) SNAPSHOT MX/TXT BEFORE ANY EDIT (D-05 safety baseline). On the Windows executor, run and SAVE
        the output of these into the plan SUMMARY so Task 3's post-edit re-check can diff against it:
        PowerShell:
          Resolve-DnsName michellengo.net -Type MX
          Resolve-DnsName michellengo.net -Type TXT
          Resolve-DnsName michellengo.net -Type CAA
        Git Bash:
          dig michellengo.net MX +short
          dig michellengo.net TXT +short
          dig michellengo.net CAA +short

    Do NOT proceed to Task 2 until (a)–(e) are confirmed done and the MX/TXT/CAA snapshot is recorded.
